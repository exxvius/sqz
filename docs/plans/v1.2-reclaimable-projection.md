# v1.2 Plan — Reclaimable-Space Projection

**Status:** Draft for review · **Target:** v1.2 (incremental, on current architecture)
**Audience:** home-media hoarders (Plex/Jellyfin, max savings, zero risk to originals)
**Hero feature:** show *how much space a run will actually reclaim* — before it starts.

---

## 1. Problem

Today the pre-run readout on Home says only:

> **240** videos · 1.4 TB

That answers "how much is queued," not the question a hoarder actually asks:
**"How much space do I get back, and how much of this is a waste of time because
it's already efficient?"**

sqz already has everything needed to answer that — it just doesn't surface it:

- The **manifest DB** (`files` table) stores realized `saved_bytes`, `size`,
  `src_codec`, and `height` for every file it has ever encoded. That is a
  ready-made corpus of *actual* savings ratios on this user's own library.
- The encoder already computes **bits-per-pixel** and knows which sources it
  would skip as `already_efficient` / `marginal`.

The hero feature turns that latent data into a projection:

> **240** videos · 1.4 TB → **~610 GB reclaimable** · 58 files skipped as already-efficient

---

## 2. Design: two-tier projection

Accuracy requires knowing each file's codec + resolution, which needs an
`ffprobe` pass. To keep the UI instant we compute in two tiers.

### Tier 1 — instant (no probe)
Runs off the existing `scan_inputs` result. Multiplies total candidate bytes by
the **global historical savings ratio** from the manifest (`SUM(saved_bytes) /
SUM(size)` over `done` rows). Shows immediately with a "~estimate, refining…"
affordance. If the manifest has no history yet, fall back to a conservative
static prior per target codec (documented constants, clearly labeled "rough").

### Tier 2 — refined (background probe)
Spawns a bounded, cancellable probe pass over the candidate files:

1. `ffprobe` each file → `(src_codec, height, bytes)`. Parallelized with a
   worker cap (reuse the existing worker-count semantics); fully cancellable if
   the input set changes.
2. Bucket each file by `(src_codec, height_band)` where `height_band ∈ {≤720,
   1080, 1440, 2160, >2160}`.
3. For each bucket, look up the **per-bucket historical ratio** from the
   manifest; blend toward the global ratio when a bucket has few samples
   (shrinkage — see §4).
4. Apply the same **skip predicates the encoder uses** (already-efficient
   codec at low bpp, marginal-bpp gate, DV-skip, sub-min-savings) to flag files
   that will almost certainly be *skipped*, and exclude their bytes from the
   reclaimable total.
5. Emit a refined projection + a per-bucket breakdown.

Tier 2 replaces the Tier 1 number in place when ready. The whole thing is
advisory — it never changes what a run does, so there is zero data-safety
surface.

## 3. Backend changes (Rust)

### New command: `project_reclaim`
```rust
#[derive(Serialize)]
pub struct ReclaimBucket {
    pub src_codec: String,     // "h264", "hevc", …
    pub height_band: String,   // "1080", "2160", …
    pub files: u32,
    pub candidate_bytes: u64,
    pub est_reclaimable_bytes: u64,
    pub est_skipped_files: u32,
    pub sample_size: u32,      // history rows backing this bucket's ratio
    pub confidence: f32,       // 0..1 from sample_size (see §4)
}

#[derive(Serialize)]
pub struct ReclaimProjection {
    pub tier: u8,              // 1 = instant, 2 = refined
    pub candidate_files: u32,
    pub candidate_bytes: u64,
    pub est_reclaimable_bytes: u64,
    pub est_skipped_files: u32,
    pub buckets: Vec<ReclaimBucket>,
    pub based_on_history_rows: u32,
}
```

- **Tier 1** variant returns immediately (`tier = 1`, no buckets) — cheap query.
- **Tier 2** is streamed/awaited; emit a `sqz-projection` event when ready so the
  UI updates without blocking (mirrors the existing event pattern in
  `events.rs`).

### Manifest queries (`core/manifest.rs`)
Two new read-only methods, both parameterized:
```rust
fn global_savings_ratio(&self) -> rusqlite::Result<Option<(f64, u32)>>;   // ratio, n
fn bucket_savings_ratios(&self)
    -> rusqlite::Result<Vec<(String /*codec*/, i64 /*height*/, f64 /*ratio*/, u32 /*n*/)>>;
```
Both filter to `status = 'done'` with `size > 0`. Height is bucketed in Rust, not
SQL, to keep the band logic in one place. **No schema change** — all columns
already exist.

### Reuse, don't duplicate, the skip predicates
The bpp / marginal / already-efficient logic lives in the encode path. Extract
the pure decision (given `codec`, `height`, `bytes`, `duration?`) into a small
`core/estimate.rs` helper that both the encoder and the projector call, so the
projection can never drift from real run behavior. This is the one refactor with
teeth — worth doing to keep the promise honest.

### Probe pass
Reuse `core/probe.rs`. Add a bounded-concurrency `probe_many(paths, cap,
cancel_token)` that returns `(codec, height, bytes)` per file and bails on
cancel. No new deps.

## 4. Confidence & honesty (don't overstate)

A projection that lies erodes trust faster than no projection. Rules:

- **Shrinkage:** a bucket's ratio blends with the global ratio weighted by its
  sample size: `w = n / (n + K)`, `K ≈ 20`. Small buckets lean on the global
  prior instead of a noisy local one.
- **Confidence surfaced in UI:** `confidence = n / (n + K)` per bucket; the
  aggregate shows a coarse band (Low / Fair / Good) rather than false precision.
- **Cold start:** with little/no history, show the number as a clearly-labeled
  rough estimate from static priors, and let the projection tighten as the user
  runs more encodes (self-improving — a nice hoarder story).
- Always phrase as **"~"** and a range, never a hard figure.

## 5. Frontend changes (TS/React)

- **`src/lib/types.ts`** — add `ReclaimBucket`, `ReclaimProjection`.
- **`src/lib/api.ts`** — `projectReclaim(inputs)`; subscribe to `sqz-projection`.
- **`HomeView.tsx`** — extend the existing scan readout (line ~452):
  - Tier 1 lands with the scan: `240 videos · 1.4 TB · ~610 GB reclaimable`.
  - Tier 2 refines in place and unlocks a **"Details"** disclosure showing the
    per-bucket table (codec × resolution → files, reclaimable, skipped,
    confidence). Reuse the existing `Collapsible` + `.bar` components; the
    glassy bars from v1.1.1 visualize each bucket's savings share.
- **Skipped-as-efficient** count gets a subtle callout so the user understands
  why candidate ≠ reclaimable (this is the "not wasting your time" moment).
- Respects reduced-motion; no layout shift (reserve space for the projection line
  so the Tier 1 → Tier 2 swap doesn't jump — CWV/CLS discipline).

## 6. Sequencing (small, reviewable PRs)

1. **`core/estimate.rs` extraction** — pull skip/bpp decision out of the encoder
   behind a pure function; unit-test parity with current behavior. *(No user-visible change; safe refactor.)*
2. **Manifest ratio queries** — `global_savings_ratio`, `bucket_savings_ratios`,
   with unit tests over a seeded in-memory DB.
3. **`project_reclaim` Tier 1** — command + wire into Home readout.
4. **Probe-many + Tier 2** — background refine, `sqz-projection` event, buckets.
5. **Home "Details" breakdown UI** — bucket table + bars + confidence.
6. **Polish + copy + screenshots** — regenerate `docs/images`, update README.

Each step ships independently; the release can cut after step 3 if needed and
still deliver the hero line.

  shrinkage blend, estimate-parity with the encoder's real skip decision
  (rstest cases per codec/band).
- **Rust:** `probe_many` cancellation + concurrency cap.
- **TS:** projection formatting, Tier 1→2 swap, cold-start labeling.
- **Visual:** add a Home scene variant to the screenshot harness with a populated
  manifest so the projection line + breakdown render deterministically; capture
  light + dark.

## 8. Explicitly out of scope (→ v2.0)

Adaptive per-title CRF **search** · content-aware codec selection · Pause/Resume
& idle scheduling · per-folder profiles · VMAF quality gate · full UX/onboarding
redesign. This plan deliberately reuses history instead of doing trial encodes,
which is what keeps it a v1.2.

## 9. Open questions for sign-off

1. **Probe cost at hoarder scale** — for 5–10k files an `ffprobe` sweep is
   non-trivial. Options: (a) probe all with a progress bar, (b) probe a capped
   sample and extrapolate per bucket, (c) cache probe results in the manifest
   keyed by path+mtime so re-scans are instant. I lean **(c) + (a)** — cache
   makes the second scan free. Agree?
2. **Cold-start static priors** — acceptable to ship conservative documented
   constants (e.g. h264→AV1 ≈ 55–65% at 1080p) for users with no history, or
   should the projection stay hidden until there's real data?
3. **Cut depth** — is the per-bucket "Details" table in-scope for v1.2, or is the
   single headline reclaimable number enough to ship first?
